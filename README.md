## 專案定位

一個純前端、免建置的檔案管理介面。使用者可以新增、瀏覽、分類篩選與刪除「超自然現象檔案」，並即時看到統計數據。介面刻意設計成機密調查文件的樣貌——掃描線、雜訊顆粒、REC 錄影指示燈、檔案編號與 `TOP SECRET` 印章——用一個有記憶點的主題,把 CRUD、表單驗證、狀態管理、無障礙這些基本功整合在一起呈現。

**為什麼選這個題目來當作品集**：它足夠小,小到可以把每一行都打磨到「說得出為什麼這樣寫」;又足夠完整,涵蓋了前端日常最核心的幾件事——資料流、元件狀態、表單驗證、效能(computed 快取、lazy load)、無障礙、以及可維護性(集中式常數管理)。

## 線上預覽與執行

本專案為純靜態網頁,**零相依套件、零建置步驟**。

```bash
# 方法一：任一本機伺服器（建議，確保字型與圖片資源正常載入）
python3 -m http.server 8000      # 然後開啟 http://localhost:8000
# 或
npx serve

# 方法二：直接用瀏覽器開啟 index.html
```

> Vue 3 透過 CDN 載入、字型使用 Google Fonts,故執行時需連網。

## 技術棧與取捨

| 項目 | 選擇 | 決策理由 |
| --- | --- | --- |
| 框架 | Vue 3 Composition API(CDN) | 題目規模不需要打包工具鏈;免建置讓 reviewer `git clone` 後能立刻開啟看到成果 |
| 樣式 | 原生 CSS + CSS Variables | 用設計 token(`--nv-green`、`--radius-md`…)集中管理視覺,不引入框架也能維持一致性 |

| 建置 | 無 | 刻意保持「開啟即用」,降低 reviewer 的閱讀成本 |

## 我在這個專案裡的工程決策

這一節是我認為作品集最該被看到的部分——**不是「做了什麼」,而是「為什麼這樣做」。**

### 1. 用 `computed` 取代方法呼叫,換取快取與更好的複雜度

各分類的筆數,最初可以寫成 `countByCategory(catId)` 方法在模板裡呼叫。但 Vue **無法快取方法的回傳值**,每次 re-render 都會重算,而且每個分類各跑一次 `filter`,複雜度是 `O(n × 分類數)`。

我改用 `categoriesWithCount` 這個 `computed`,以一次 `reduce` 走訪算完所有分類筆數:

```js
const categoriesWithCount = computed(() => {
  const countMap = items.value.reduce((acc, it) => {
    acc[it.category] = (acc[it.category] || 0) + 1;
    return acc;
  }, {});
  return categories.map((cat) => ({ ...cat, count: countMap[cat.id] || 0 }));
});
```

`items` 沒變就不重算(快取),複雜度降為 `O(n)`,模板端也從呼叫函式變成宣告式地讀 `cat.count`。

### 2. 消滅魔術數字:集中式 `LIMITS` 常數

字數上限 `300`、警示閾值 `80`、截斷長度 `50`… 這些數字原本散落在 `maxlength`、`placeholder`、字數計數、變色邏輯多處。**只要改一個地方就會漏改其他地方**。我把它們收斂成單一來源:

```js
const LIMITS = {
  NAME_MAX: 30,
  DESC_MAX: 300,
  DESC_NEAR: 240,
  TRUNCATE_LEN: 50,
  HIGH_RISK_RATING: 4,
};
```

模板全部綁定 `LIMITS`,未來調整限制只需動一處,所有相依處自動同步。

### 3. 持久化時剝除 UI 狀態

`expanded`(是否展開)、`imgError`(圖片是否載入失敗)屬於暫時性 UI 狀態,存進 `localStorage` 沒有意義還浪費空間。寫入前用解構把它們濾掉,只留下真正該持久化的資料欄位:

```js
const persistData = items.value.map(
  ({ expanded, imgError, ...rest }) => rest
);
```

> 命名補充:這裡的 `rest` 我後來認為不是好名字——它和常見的重置方法 `reset` 只差一個字母,容易讓讀者誤以為是函式。**好的變數名應該讓不熟語法的人也能猜對意圖**,更貼切的命名會是 `persistData` / `storable`。這個反思本身就來自一次和 AI 的辯論(見下一節)。

### 4. 生命週期成對管理,杜絕記憶體洩漏

時鐘用 `setInterval` 每秒更新,在 `onMounted` 開啟、就一定在 `onUnmounted` 用 `clearInterval` 收掉。有開必有關,避免元件卸載後計時器仍在背景累積。

### 5. 韌性優先的圖片處理

`loading="lazy"`(HTML 原生)讓離屏圖片延後下載;`@error`(綁定 HTML 原生 error 事件)在圖片壞掉時切換成 `NO VISUAL RECORD` 佔位,而不是留下破圖。外部圖床不可控,前端要為「資源失敗」預留退路。

## 無障礙（WCAG A + AA）

無障礙是這個專案從規劃階段就納入、而非事後補上的需求:

- **色彩對比**:所有文字 / 背景經驗證 ≥ 4.5:1(夜視綠對深底約 14:1)。
- **星等評分**:以 `role="radiogroup"` + `aria-checked` 實作,並依 W3C 標準用 roving `tabindex`——只有選中項可被 Tab,群組內用方向鍵切換。
- **表單錯誤**:`aria-invalid` + `aria-describedby` 把錯誤訊息與字數計數關聯到欄位,螢幕閱讀器聚焦時會一併讀出。
- **語意化與隱藏標題**:統計區用 `visually-hidden` 的 `h2` + `aria-labelledby`,視覺不佔位、輔助技術仍讀得到區塊名稱。
- **焦點樣式**:保留並自訂明顯的 `:focus-visible` 樣式,不移除 outline。
- **動畫友善**:`prefers-reduced-motion` 下關閉閃爍與掃描動畫。

## 我如何運用 AI 協作開發

我把 AI(Claude)當作**結對程式設計的夥伴**,而不是自動產碼機。完整的協作過程保存在 [`AI對話紀錄.md`](./AI對話紀錄.md)。我的協作原則有三:

### 一、先規劃、後實作——不讓 AI 一次噴出全部程式碼

我要求 AI 先進入「規劃階段」,產出整體架構、互動流程、輸入 / 輸出欄位清單與規格衝突點,**經我確認後才寫 code**。這讓我在動手前就掌握全貌,也讓每個決策(分類方式、是否加 localStorage、圖片欄位是否必填)都是我拍板的,而不是被預設值決定的。

### 二、逐行理解,不接受「看起來會動」的程式碼

對話紀錄裡有大量我針對單一屬性、單一語法的追問——`aria-labelledby` 的用途、`padStart` 的行為、`onMounted` vs `onUnmounted` 的差異、`@error` 到底是 Vue 還是 HTML 的東西。**我不會把不理解的程式碼放進專案。** 每一段最終留下來的 code,我都能說明它為什麼在那裡。

### 三、質疑 AI、並主導更好的決策

這是我認為最關鍵的部分。AI 會出錯、會含糊,而我的價值在於能判斷並修正它:

- **我推翻了 AI 的架構寫法**:是我主動提出「分類筆數為什麼不用 `computed` 包起來」,促成第 3 節那次重構——由我發現問題、AI 補上實作。
- **我糾正了 AI 不嚴謹的說法**:AI 一開始說變數命名 `rest` 是「社群慣例」。我查了 MDN 與技術文章後指出,官方範例根本用不同名字(`theArgs`、`z`),`rest` 並非任何規範或約定。AI 承認並收回了說法。接著**我進一步提出**:`rest` 和重置方法 `reset` 只差一個字母,會誤導讀者——這個命名原則後來成了我改善程式碼的依據。
- **我要求把限制抽成常數**:是我指出「300 這個數字寫死在多處,改的時候會漏」,才有了集中式 `LIMITS`。

> **我對「用 AI 寫 code」的態度**:AI 大幅加快了我把想法落地的速度,但**工程判斷、架構決策、可維護性與正確性的最終責任在我**。我用 AI 來加速探索與實作,而不是外包思考。這份對話紀錄的價值,不在於證明我會用 AI,而在於呈現我**在與 AI 協作時仍持續做工程判斷**。

## 已知事項與後續規劃

保持誠實,以下是我清楚知道、若專案續作會處理的項目:

- **localStorage 讀取目前為關閉狀態**:`loadData()` 中讀取本機資料的那行暫時以 `const raw = null` 停用,因此每次重整都會回到預設種子資料(寫入邏輯仍正常)。啟用只需還原為 `localStorage.getItem(STORAGE_KEY)`。
- **後續規劃**:抽離元件(卡片 / 表單 / 統計)、導入建置工具與單元測試(Vitest)、以 `<transition-group>` 為列表增減加上動畫、支援檔案匯出 / 匯入。
